The issue: JavaScript

Simply put, not everyone wants to use JavaScript.

Some users have safety concerns.

Some developers have bloat concerns.

The solution: what we already have!

The solution is simple - a radio button state machine!

Or, in non technical terms, introducing a little bit of boilerplate to the markup and some very simple CSS in order to get this very presentation to work!

The message: stop and think!

The moral, or the message, is to not reach for JavaScript!

Modern HTML/CSS is powerful and fun, there's a good chance you don't need JavaScript.

Sure, for complicated apps it's necessary, but you can build some sophisticated stuff without it! Finding a HTML/CSS alternative will be more performant and allow your app to work in environments where JavaScript isn't avaliable.

In a similar spirit to the gemini protocol, help keep the web as 1.0 as possible!

Thank you for reading!

Nice presentation, how does it actually work?

It works on a radio button based state machine.

CSS doesn't really do "state"

We can't keep track of a current_slide index or anything.

What we can do however is check whether a input[type="radio"] is checked or not.

The state machine

Each slide is placed after a input[type="radio", name=*]. If this radio is :checked, then the slide is shown.

The next/previous input element is then positioned on the left or the right, relative to the :checked one.

Each presentation requires a unique name to use in the input tags - or else it gets all confused.

The actual implementation

Because CSS is weird, we have to implement the idea like this:

  1. Check if a given input[type="radio", name=*] has been checked

    2. If it has, then we make the sibling section (+ sibling) visible.

  2. Check if the next input[type="radio", name=*] is checked

    3. This sets up positioning for the left hand side selector (the "previous slide" button)

  3. Display the next input[type="radio", name=*]

    4. This sets up positioning for the right hand side selector (the "next slide" button)
